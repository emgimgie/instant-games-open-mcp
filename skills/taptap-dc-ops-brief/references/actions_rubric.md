# 点赞/回复动作建议 Rubric（简化版）

本文件用于在“只读简报”之外，给出可控的运营动作建议。任何写操作都必须先征得用户许可。

## 点赞（like）建议

优先点赞的评价类型：

- 信息密度高：包含明确问题描述、复现步骤、机型/版本信息
- 具有代表性：反映当前版本最常见的 1-2 个核心问题
- 正向长评：能帮其他用户做决策，且语气友好

不建议点赞的情况（除非用户明确要求）：

- 明显引战、辱骂、人身攻击
- 涉及敏感/合规风险（泄漏、外挂、灰产）

## 官方回复（reply）建议

建议回复的触发条件（满足任一即可纳入候选）：

- 负向或中立，且包含可行动点（崩溃、卡顿、登录、支付、内容违规等）
- 互动较高（点赞/回复多）或对转化影响大（新手关卡、下载/安装、付费节点）
- 同类问题集中出现（“连着 3 条提同一个点”）

不建议直接回复的情况（可先内部确认）：

- 涉及账号/支付隐私，需要引导走客服渠道
- 需要承诺时间点或赔付（除非用户提供明确口径）

## 回复草稿模板（短句、可落地）

结构建议（尽量 2-4 句结束）：

1. 致谢 + 共情（不辩解）
2. 问题归因或进展（只写确定信息）
3. 临时方案/下一步（给可执行动作）
4. 补充信息收集（可选，最多 1 个问题）

示例（性能/卡顿）：

- “感谢反馈，我们已关注到近期部分机型的卡顿问题，团队正在排查。你可以先尝试关闭高画质/重启游戏看看是否缓解；如果方便，告知一下机型和系统版本，我们会更快定位。给你带来不便非常抱歉。”

示例（功能 bug）：

- “感谢指出，这个问题我们已经记录并在修复中。麻烦你描述下触发步骤（例如在第几关/哪个入口出现），我们会优先跟进处理。”

## 执行前的用户确认（硬约束）

调用 MCP 写操作前，必须把这三项展示给用户并询问是否执行：

- 目标评价：`review_id` + 评价内容摘要
- 拟执行动作：点赞/回复
- 拟回复文本（如为回复）
